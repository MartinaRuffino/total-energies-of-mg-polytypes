LMTO package (v7.11)

This package is a collection of electronic structure codes based mostly on the local-density approximation (LDA) to density-functional theory (DFT) to solids. Many implementations of DFT exist. This one is an all-electron kind, similar in many respects to WIEN 2K, Exciting, and Fleur. The codes have been written mainly by M. van Schilfgaarde (Mark.vanSchilfgaarde@gmail.com), though many people have made important contributions.

Unique Features : The basis set used to represent crystal eigenfunctions is composed of atom centered functions (linear muffin tin orbitals) --- rather than the more commonly used augmented plane wave basis. This has advantages --- basis sets are much smaller for a given level of accuracy, but it also requires somewhat more knowledge for the user to operate. It is also possible to take plane waves and atom centered functions in combination -- another unique feature of this package.

Another unique feature is the augmentation. It is carried out in a manner somewhat resembling the PAW method, though it has a proper convergence that all-electron methods possess.

The package contains a variety of useful special purpose programs, e.g. ability to calculate magnetic exchange interactions, and an implementation of the coherent potential approximation.

Perhaps the most important is the connection to implementation for GW calculations, in an all-electron framework. GW is usually implemented as an extension to the LDA, i.e. G and W are generated from the LDA. You can use this package for LDA-based GW calculations, but it also implements the Quasiparticle Self-consistent GW approximation (QSGW). QSGW is novel theory, which surmounts many of the well-known failings in density-functional theory (DFT), and LDA-based GW. It is based on the GW approximation, but does not use the DFT Green's function as a starting point. QSGW may be thought of as an optimised form of the GW approximation of Hedin. While QSGW is vastly superior to ordinary GW, it is much more demanding computationally.

To see some of the features the package offers, take a look at the demo that works through a density-functional calculation of Si, followed by one-shot and QSGW calculations. See also the demo that works through a self-consistent LDA calculation of the magnetic material Co3Pt, followed by a calculation of the magnetic exchange parameters.

Introduction

Density functional theory is documented in many books. A particularly good one, with many details explaining how is implemented, is Richard Martin's Electronic Structure. Some lecture notes provide a brief outline to how DFT is implemented, and how the techniques of this package are connected to implementations of DFT in general. Another set of lecture notes offers a brief explanation of the limitations of density functional theory, and a brief introduction to the Quasiparticle Self-consistent GW approximation. This set offers practical demonstrations of some capabilities in these programs. It shows you how to carry out these demonstrations on your own.

Several LDA codes belong to the package: the standard one is an all-electron full-potential method; there is also a molecules-specific code, and several codes that make the Atomic Spheres Approximation to the LDA (ASA), a theory formulated by O. K. Andersen. Development of the ASA occured historically; it is still of value because, while being less precise than what full-potential methods offer, it is extremely efficient and many properties can be calculated with it, such as optical properties and magnetic exchange parameters entering into the Heisenberg model.

These codes use the method of linear muffin-tin orbitals (LMTOs) formulated by O. K. Andersen within the ASA. LMTO refers to a basis set; it often is, but need not be, tied to the ASA. There is also a full-potential implementation in this package. While the envelope functions (basis) are related to LMTOs, they are actually convolutions of Gaussian and Hankel functions, and are more flexible and general. They are also smooth and regular at the origin --- important properties in the full-potential context, but the greater flexibility also adds greater complexity. Starting with v7, the basis can be built out of a combination of Augmented Plane Waves (APWs) and LMTOs (the PMT method), a capability unique to this code.

This package also contains front-end for GW calculations, an all-electron implementation of conventional GW where G and W are generated from the LDA, and also the Quasiparticle Self-consistent GW approximation (QSGW). The GW approximation is a perturbation theory around some one-body hamiltonian (e.g. the LDA hamiltonian), based on many-body perturbation theory. In the QSGW approximation, self-consisistency is used to minimize the explicitly many-body parts of the hamiltonian, putting as much as possible into the one-body part. This makes it very accurate, much superior to conventional implementations of GW that usually rely on the LDA for the starting point.

There is also an implementation for empirical tight-binding hamiltonians --- useful for simulations of systems too large to be easily accessible by density-functional methods.

Capabilities

This package has several kinds LDA-based programs designed for different purposes, outlined below. In the LDA (and other theories) the effective potential electrons see is generated self-consistently in some approximation to the exact many-body problem. The effective potential of a system must be determined iteratively in a self-consistent manner. What form self-consistency takes depends on the program used and property considered. After self-consistency is reached, programs can be used to probe for a variety of physical properties, such as the energy bands, density-of-states, optical properties, magnetic exchange interactions, and Landauer-Buttiker transport through layered systems. Recently the ability to calculate properties alloys and disordered spin systems in the coherent potential approximation has been added.

Electronic structure theory is implemented in the following forms:

Atomic Spheres Approximation  There is a suite of codes based on the ASA. The ASA code is very fast and efficient, and it has many extensions, e.g. noncollinear magnetism, optics, and a scheme to fit bands or densities-of-states to given quantities (e.g. those generated by QSGW), by adjustments to ASA-LDA potential parameters via a Levenberg-Marquardt minimization algorithm. The user is cautioned that the ASA is much less reliable, and must be used with some care. For a quick look at how it works, skim through this demo of the ferromagnet Co3Pt.
lmto.html describes the capabilities of ASA code. It contains information generic to the package as a whole, such as a description of a number of useful auxiliary programs, such as a supercell maker. Even if you do not plan to use the ASA package, it is recommended that you read this document.

Full-potential LDA  fp.html documents the capabilities of the all-electron, full-potential LDA program. It implements standard DFT functionals, though the form of implementation is novel. It has a three-fold representation of the charge density and has shares elements in common with the LAPW and PAW methods, and may be thought of as a framework that bridges the two.

All-Electron GW  This code is also linked to a GW package; For a quick look at a practical demonstration of the LDA and GW package working together, see the LDA + GW + QSGW demo for Si. The driver to the GW package is documented in gw.html,. See this page for a tutorial outlining how to construct the input to the GW package, and the linkage between the LDA and GW packages.

Molecules code  There is a program designed just for molecules. It is fast and efficient, but a little unstable. It is not documented. If you want to use this package, contact Tony Paxton at Tony.Paxton@kcl.ac.uk.

Empirical tight-binding  is not density-functional based, but implements model tight-binding hamiltonians. The user can specify the form of the model in one of several possible ways. This code has been highly tuned and it can be used to efficiently study systems with thousands of atoms. To use this code contact the primary author of the current implementation, Dimitar Pashov (dimitar.pashov@kcl.ac.uk).

These various implementations are collected in a family of executable programs that share a common, elegant input system, which has some elements of a programming language. The family consists of the following:

  • lm: the standard LDA band program, in the Atomic Spheres Approximation, and its companion lmstr, which makes structure constants lm requires to execute. There is a tutorial that goes through a sample calculation, as well as a demo that highlights some of its features.
  • lmf: the standard full-potential LDA band program, and a companion program lmfa for free atoms (needed to start lmf). Both the basis and the form of implementation of density-functional theory are original with this code; a reference describing the implementation can be found in this book chapter. The theory for the envelope functions can be found in this J. Math Phys. article. One tutorial explains the input and output; another shows to build input files automatically.
  • lmgf: a Green's function program based on the ASA. Its unique contribution to the suite is that it permits the calculation of magnetic exchange interactions, and (starting with v7.8) has an implementation of the coherent potential approximation to treat chemical and/or spin disorder.
  • lmpg: a layer Green's function program also based on the ASA that works for layered systems (periodic in 2D). It can calculate transport using the Landauer-Buttiker formalism, and has a non-equilibrium capability. There a tutorial, though it is somewhat out of date.
  • lmfgwd: a driver for an all-electron GW package, which has a highly robust product basis for the construction of the dielectric function and coulomb interaction. It can be run in 1-shot mode, or the quasiparticle self-consistent mode.
  • lmmc: a (fast) LDA-based molecules program (not documented).
  • tbe: a band structure program that uses empirical tight-binding hamiltonians. One unique feature of this package is that self-consistent calculations can be done (important for polar compounds), and includes Hubbard parameters. It is also highly parallelized, and versions can be built that work with GPU's.

    • The following executables are compiled with the basic package:
  • blm: an input file maker for the full-potential program lmf; see Building_FP_input_file.html for a tutorial.
  • lmscell: a supercell maker.
  • lmchk: a neighbor table generator and augmentation sphere overlap checker. There is an option to automatically determine sphere radii. (There is another option to locate interstitial sites where empty spheres or floating orbitals may be placed --- important for ASA and some GW calculations).
  • lmxbs: generates input for the graphics program  xbs  written by M. Methfessel, which draws pictures of crystals.
  • lmdos: generates partial densities of states, run as a postprocessor after execution of lmf, lm, or tbe.

    • Some other auxiliary programs are also made; see here. To plot the bands or density-of-states you can use your favorite package, or consider using a graphics package (FPLOT.vsn.tar.gz) tailored to work with these programs. The  FPLOT  package has a number of useful features not found in widely used packages, such as the ability to plot energy bands with color weights.

    Input System

    All programs use a common input system. It is a unique system that parses input in a largely format-free, tree-structured format. There is some programming language capability: e.g. input lines can be conditionally read, you can declare variables and use algebraic expressions. Thus a single file can serve as database (one input file for many materials systems) and for documentation. Look at input-file-style.html for an easy-to-follow example illustrating how an input file is structured, and how input is organized by categories and tokens. It is an operational input file, and you can see how quantities used in typical calculations are supplied; and it hints at some of the many possibilities of this very flexible and versatile interface. You are advised to go through that web page before attempting any calculations. input.pdf provides a somewhat dull, but more complete description of the input syntax. Finally see tokens.html for documentation of tokens read by the various programs, and their meaning.

    There is one main input file all programs use,  ctrl.ext; it can be complicated as there are many options available. Building_FP_input_file.html is a tutorial that explains options automatically build an input file for the FP program, starting from lattice data you supply or import.

    A second stream of input comes through command-line switches. There are a few switches all programs use, and there are also a large number of program-specific options, described here. You can get a listing of switches a particular program uses by invoking the program with  --help, e.g.  lm --help.

    There can be many files associated with each calculation. Most of them are made automatically by an executable program; some files are editable so the user can modify them. In a few instances more than one input file is required to get started, e.g. the GW code requires file  GWinput. File  README  supplies a brief description of the contents of each file in this directory, and the directories in the top-level directory (note: this file is a bit out of date).

    Backwards Compatibility

    Version 7 underwent a radical internal redesign of the input system. Externally there are only a few differences; but the v7 input files are not exactly backwardly compatible with the v6 files. A few tokens were put in different categories; some default values were changed; some tokens were merged or grouped a little differently. Also the old style inputs of complex numbers ("structures"), e.g.  MSTAT  and  SDYN, are now rendered in a (consistent) tree format.

    Program lm67 can facilitate the job of updating the  ctrl file. It will read v6 input files and print to standard output changes you need to make to render a v6 input file compatible with v7 input. The needed changes depend to some extent on the executable program. By default lm67 looks for changes appropriate to lmf. For changes relevant to other executables, e.g. lm, do: 

      lm67 --prog=LM ...
    Besides rearrangements or renaming of some tokens, one or two small differences in the computational part were made to the FP program lmf, and results may be slightly different from v6.

    Installation

    Starting with version 7.11, the installation scripts have also been radically revised. The entire package is distributed as one tarball; executables are built with a python script.

    Old style installation

    Prior to version 7.11, the package was distributed in parts: 
    TB.vsn.tar.gz OPTICS.vsn.tar.gz MOL.vsn.tar.gz GF.vsn.tar.gz ASA.vsn.tar.gz SX.vsn.tar.gz PGF.vsn.tar.gz NC.vsn.tar.gz GWD.vsn.tar.gz  FP.vsn.tar.gz
    in addition to the math library  SLATSM.vsn.tar.gz which was installed independently. In this scheme, executables are installed from a  configure  script, which creates a collection of Makefiles. Executables are built in the top-level directory. To make an easy transition, compatibility the old style installation scripts are temporarily maintained. Executables are built in the top-level directory. To install you must first install package  SLATSM.70.tar.gz  (or a later version),  a library of generic numerical, string and functions. The  SLATSM  library comes with its own set of installation instructions (file startup/README), which must be done before installing this package.
